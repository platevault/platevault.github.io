# PlateVault Docs

Source for the [PlateVault](https://github.com/platevault/platevault) documentation site, built with [Astro Starlight](https://starlight.astro.build); it lives in its own repo (separate from the app monorepo) so it can move independently without dragging the app's history along.

This repo is named `platevault.github.io`, so GitHub Pages serves it at the org
root — <https://platevault.github.io> — and `base` is `/`. Internal links in
content stay relative (`/manual/...`, `/how-to/...`); the Pages `site`/`base`
URL is hardcoded only once, in `astro.config.mjs`. A future move to a custom
domain changes `site` and leaves everything else alone.

## Development

```
pnpm install
pnpm dev       # localhost:4321
pnpm build     # -> dist/
```

Content lives under `src/content/docs/` (`manual/` and `how-to/` sections);
each page currently has a `WRITER TODO` HTML comment pointing at the alm
repo's `docs/journeys/JXX` ground truth it should be written from — content
authoring is a separate pass, not part of this scaffold. Placeholder
screenshots (`src/assets/screenshots/*.svg`) are generated by
`scripts/generate-placeholders.mjs`; track outstanding captures in
`src/content/docs/reference/screenshots-todo.md`.

## Deploys

`.github/workflows/docs.yml` builds and deploys to GitHub Pages on push to
`main`, `workflow_dispatch`, or a `repository_dispatch` event of type
`platevault-released`.

This mirrors `prompting-press/prompting-press`'s docs pipeline: there,
`release.yml` has a `docs-snapshot` job that runs on a real release, freezes
versioned docs content, and merges to `main`, which push-triggers `docs.yml`
to build and cross-repo-publish to `prompting-press.github.io`'s own
Pages-serving workflow. This repo has no code to snapshot per version and
already **is** the Pages target, so the shape simplifies to: build+deploy
in-repo, triggered by push to `main` or a `repository_dispatch` from the
`alm` release.

**Follow-up needed in `alm` to wire up the release trigger** (not added
here — this repo's workflow only listens for it): add a step to `alm`'s
`.github/workflows/release-please.yml`, after a release is published, that
fires the dispatch:

```yaml
- name: Notify docs repo of release
  if: steps.release.outputs.release_created == 'true'
  run: |
    gh api repos/platevault/platevault.github.io/dispatches \
      -f event_type=platevault-released \
      -f "client_payload[version]=${{ steps.release.outputs.tag_name }}"
  env:
    GH_TOKEN: ${{ secrets.SOME_TOKEN_WITH_DISPATCH_SCOPE }}
```

Until that's added, docs deploys happen on every push to `main` here plus
manual `workflow_dispatch` — release-driven refresh (e.g. of version strings
or download links once the writer pass adds them) is not yet automatic.
